# CERTIAURA CONTINUITY AND CHANGE CONTROL

**Version:** 1.0.0  
**Status:** Active Reference Rule

## Purpose

Prevent time being lost through:

- Recommending systems that already exist;
- Reopening settled decisions without cause;
- Creating duplicate governance documents;
- Quietly changing previously agreed architecture;
- Treating proposals as though they were approved decisions.

## Required check before proposing change

Before proposing a new structure, check:

1. Does an agreed standard already cover this?
2. Is this already in the Decision Log, Master Asset Register, repository standards or Master Project Charter?
3. Is there a real production problem that the current system cannot solve?
4. Is the proposed change an extension, amendment, replacement or genuinely new requirement?

## Change categories

### Extension
Adds capability without changing the existing decision.

### Amendment
Modifies part of an existing decision.

### Supersession
Replaces an existing decision.

### New decision
Covers a subject not already governed.

## Required form for material change proposals

A material proposal should state:

- **Existing decision:** what is already agreed
- **Problem:** why it is insufficient
- **Proposed change:** what should change
- **Impact:** what files, systems or assets are affected
- **Decision type:** Extension / Amendment / Supersession / New decision

No material change should be presented as though it were already approved.

## Production-first default

When no genuine gap exists:

> Continue production using the existing system.

Do not create another charter, dashboard, decision log, asset register, naming standard or repository framework merely because a differently worded version could be created.

## Source priority

When resolving continuity:

1. Latest explicit founder instruction;
2. Master Project Charter;
3. Locked Decisions Reference;
4. Existing governance standards and Decision Log;
5. Current repository implementation;
6. Earlier chat discussion not yet formalised.

Where uncertainty remains, preserve the existing implementation and ask for a targeted decision rather than redesigning the system.
